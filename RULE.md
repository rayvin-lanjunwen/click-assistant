# Click Assistant 项目执行规则

文档版本：v1.0.0
最后更新：2026-07-08
维护人：项目负责人

本文档是 Click Assistant 项目内部执行规则的最高优先级说明。每次处理项目任务时，应优先读取并遵循本文档中的规则。

对 `RULE.md` 本身的任何修改，必须在 `CHANGELOG.md` 中以 `docs:` 类型记录变更内容，并经项目负责人审核同意后方可提交。

## 1. 规则优先级

- `RULE.md` 是项目内部执行规则中优先级最高的文件。
- 每次开始工作前，必须完整读取 `RULE.md`，确认当前任务需要遵循的核心规则。
- 当其他项目文档与 `RULE.md` 出现不一致时，应优先以 `RULE.md` 为准，并在必要时记录或提示该差异。

## 2. 风格规范

- 每次处理任务时，必须读取并遵循 `STYLEGUIDE.md`。
- 后续所有文档、代码、命名、注释、提交信息和表达方式，都应尽量符合 `STYLEGUIDE.md` 中的规范。
- `STYLEGUIDE.md` 中约定的提交规范（Conventional Commits，约定式提交）和分支命名规范，在代码提交和分支管理时必须严格执行。
- 如果任务中出现新的风格约定，应优先补充到 `STYLEGUIDE.md`，确保规范可以持续沉淀。

## 3. 项目理解

- 每次处理任务前，应读取 `README.md`，用于了解项目的整体说明、目标、结构和当前状态。
- 如果 `README.md` 的内容不足以支持当前任务，应结合相关文件继续补充理解。
- 当项目结构、使用方式或核心说明发生变化时，应及时更新 `README.md`。

## 4. 工作记录

- 每次工作完成后，必须更新 `WORKLOG.md`。
- `WORKLOG.md` 应记录本次工作完成了什么、涉及了哪些文件，以及是否存在后续待办事项。
- 工作记录的具体格式要求，包括时间格式、内容粒度和排序方式，参见 `STYLEGUIDE.md` 第 3 节“工作日志规范”。
- 工作记录应保持简洁、准确，方便后续追踪项目进展。

## 5. 更新日志

- 如果本次工作涉及版本变化、功能新增、功能调整、Bug 修复或其他影响项目行为的变更，应更新 `CHANGELOG.md`。
- `CHANGELOG.md` 的具体格式要求，包括版本号格式、时间格式、分类前缀和关联引用，参见 `STYLEGUIDE.md` 第 4 节“更新日志规范”。
- 必须记录的变更包括：任何影响用户使用行为的功能变化、版本发布、架构调整和破坏性变更。
- 可以不记录的变更包括：不影响用户使用行为的内部代码重构、配置文件调整和测试用例补充。
- 仅涉及临时探索、无实际文件变更或不影响项目结果的操作，可以不写入 `CHANGELOG.md`。
- 每次正式发布版本或打 Git Tag（Git 标签）前，必须先更新 `CHANGELOG.md`，再发布。

## 6. 分支与提交规范

- 主分支：`main` 分支始终保持可发布状态，禁止直接推送代码。
- 开发分支：所有功能开发、Bug 修复必须在独立分支上进行，分支命名规则参见 `STYLEGUIDE.md` 第 2.5 节“分支命名规范”。
- 合并要求：合并到 `main` 分支必须通过 Pull Request（拉取请求），且至少由一名非本人成员审核通过。
- 提交信息：所有提交信息必须遵循 `STYLEGUIDE.md` 第 2.4 节“提交规范”，使用 Conventional Commits（约定式提交）格式。
- 禁止将包含调试代码、临时文件或未完成功能的提交合并到 `main` 分支。

## 7. Issue 与 Pull Request 规范

- 提交 Issue 时，应使用统一模板，并包含以下内容：
  - 问题描述：清晰说明遇到了什么问题。
  - 复现步骤：说明如何复现该问题。
  - 预期行为：说明正常情况下应该发生什么。
  - 实际行为：说明实际发生了什么。
  - 环境信息：说明操作系统、版本号等必要环境信息。
- 提交 Pull Request 时，应使用统一模板，并包含以下内容：
  - 关联 Issue 编号：例如 `Closes #12`。
  - 变更内容概述：说明本次 PR 做了哪些改动。
  - 测试情况：说明是否已自测，以及测试覆盖情况。
  - 破坏性变更说明：如有破坏性变更，必须明确标注。
  - 截图或录屏：涉及 UI 变更时，应附带视觉证据。
- PR 合并前，必须确保所有 CI 检查（如测试、代码检查）通过。

## 8. 版本号规范

- 项目版本号遵循 `STYLEGUIDE.md` 第 4 节中定义的 Semantic Versioning（语义化版本）规范，格式为 `主版本号.次版本号.补丁版本号`。
- 版本号变更规则：
  - 主版本号加一：存在不兼容的 API 变更或架构重写。
  - 次版本号加一：新增向后兼容的功能。
  - 补丁版本号加一：修复 Bug 或进行性能优化。
- 每个版本发布时，必须在 `CHANGELOG.md` 中记录，并在 GitHub Releases（GitHub 发布页面）中同步发布。
- 版本号以 Git Tag（Git 标签）形式标注，格式为 `vX.Y.Z`，例如 `v2.1.3`。

## 9. 规则更新

- 对 `RULE.md` 本身的修改，必须经过项目负责人审核同意后方可提交。
- 修改后必须在 `CHANGELOG.md` 中以 `docs:` 类型记录本次规则变更的内容和原因。
- 规则变更生效后，应同步知会所有项目成员。
- 当 `STYLEGUIDE.md` 或项目结构发生重大变化时，应同步检查 `RULE.md` 是否需要相应更新。
