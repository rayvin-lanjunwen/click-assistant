# Click Assistant 需求规格说明书

文档版本：v0.1.0
最后更新：2026-07-09
维护人：项目负责人
当前状态：初稿，已确认首版设计约束

本文档用于记录 Click Assistant 在规划与需求分析阶段形成的项目目标、范围、功能需求、非功能需求和验收标准。后续进入开发阶段前，应先确认本文档中的关键假设和待定问题。

## 1. 项目规划

### 1.1 项目目标

Click Assistant 旨在提供一个本地化的点击与键盘输入辅助工具，帮助用户把重复性鼠标点击和键盘连按操作配置成可复用任务，并支持按需启动、暂停、停止和查看执行结果。

项目优先解决以下问题：

- 减少重复点击和重复按键带来的手动操作成本。
- 降低需要固定输入顺序、固定时间间隔的操作出错率。
- 为测试、数据录入、流程验证等场景提供简单可靠的本地输入执行能力。

### 1.2 项目愿景

Click Assistant 应成为一个轻量、清晰、可控的桌面点击助手。它不追求复杂自动化平台的全量能力，而是优先做好“配置点击任务、稳定执行任务、随时安全停止任务”这条核心链路。

### 1.3 目标用户

- 需要进行重复点击操作的普通用户。
- 需要验证界面流程或重复执行点击场景的测试人员。
- 需要处理固定步骤、固定位置操作的办公用户。
- 需要本地、轻量、无需复杂脚本的自动点击工具使用者。

### 1.4 项目范围

MVP（最小可用版本）优先包含：

- 点击任务创建、编辑、删除。
- 输入步骤配置，包括动作类型、鼠标坐标、点击类型、键盘按键、连按次数、间隔时间和执行次数。
- 鼠标坐标采集。
- 键盘单键连按。
- 任务启动、暂停、继续和停止。
- 全局或应用内快捷键。
- 执行状态和基础日志。
- 本地保存任务配置。

暂不纳入 MVP 的内容：

- 图像识别、OCR（文字识别）或 AI 自动识别界面元素。
- 云端同步、账号体系和多人协作。
- 浏览器自动化、接口自动化或跨设备控制。
- 绕过安全验证、批量刷量、游戏外挂或未经授权控制第三方系统的能力。
- 多显示器专项适配。
- 任务导入导出。

### 1.5 首版设计约束

- 首版仅支持 Windows。
- 技术栈采用 C# + WPF，WinUI 作为后续候选方案。
- 坐标采用屏幕绝对坐标，后续再考虑窗口相对坐标。
- 必须支持全局快捷键，尤其是立即停止。
- 数据存储采用 SQLite。
- 首版不支持多显示器专项适配。
- 首版不支持任务导入导出。
- UI 风格要求优雅简洁，可参考透明毛玻璃感。

## 2. 角色与使用场景

### 2.1 普通用户

普通用户希望通过简单界面创建点击任务，不需要编写脚本，即可完成重复点击。

典型场景：

- 在固定位置重复点击按钮。
- 按固定顺序点击多个位置。
- 临时暂停任务，确认界面状态后继续执行。

### 2.2 测试人员

测试人员希望快速配置界面操作步骤，用于重复验证某个流程是否稳定。

典型场景：

- 配置一个固定点击流程，重复执行 10 次。
- 通过执行日志查看任务是否完整完成。
- 快速停止异常执行中的任务。

### 2.3 项目维护者

项目维护者需要保证功能结构清晰、配置可扩展、后续可以逐步加入更多自动化能力。

典型场景：

- 扩展新的点击类型。
- 增加新的触发方式。
- 优化任务执行和日志模块。

## 3. 功能需求

### 3.1 任务管理

- 用户可以创建新的点击任务。
- 用户可以编辑已有点击任务的名称、描述和步骤。
- 用户可以删除不再需要的点击任务。
- 用户可以复制已有任务，作为新任务的起点。
- 用户可以查看任务列表，包括任务名称、最近运行时间和启用状态。

### 3.2 输入步骤配置

- 用户可以为一个任务添加多个输入步骤。
- 每个输入步骤至少包含：步骤名称、动作类型、执行前等待时间、执行后间隔时间。
- 鼠标点击步骤应包含：点击坐标、点击类型。
- 键盘按键步骤应包含：按键名称、连按次数、连按间隔。
- 点击类型至少包含：左键单击、左键双击、右键单击。
- 每个步骤可以调整执行顺序。
- 用户可以启用或禁用某个步骤。

### 3.3 坐标采集

- 用户可以进入坐标采集模式。
- 坐标采集模式中，系统应显示当前鼠标位置。
- 用户确认后，系统将当前鼠标坐标写入指定鼠标点击步骤。
- 坐标采集过程应支持取消，避免误写入。

### 3.4 任务执行控制

- 用户可以启动指定点击任务。
- 用户可以暂停正在执行的任务。
- 用户可以继续已暂停的任务。
- 用户可以立即停止正在执行的任务。
- 系统应提供清晰的运行状态，例如：未运行、运行中、已暂停、已完成、已停止、执行失败。

### 3.5 循环与间隔

- 用户可以设置任务整体重复次数。
- 用户可以设置任务开始前倒计时。
- 用户可以设置步骤之间的执行间隔。
- 用户可以设置任务完成后的停止行为。

### 3.6 快捷键

- 用户可以使用快捷键启动、暂停和停止任务。
- 停止快捷键必须足够醒目，并优先保证可用性。
- 快捷键配置应避免与常见系统快捷键冲突。

### 3.7 执行日志

- 系统应记录任务开始时间、结束时间、执行结果和错误信息。
- 用户可以在界面中查看最近执行日志。
- 日志内容应简洁明确，不展示过多内部实现细节。

### 3.8 本地配置存储

- 点击任务和基础设置应保存在本地。
- 应支持应用重启后恢复任务列表。
- 本地配置应便于后续扩展字段。

## 4. 非功能需求

### 4.1 易用性

- 界面默认使用中文。
- 主流程应围绕任务列表、任务编辑和运行控制展开。
- 常用操作应尽量在主界面完成，减少多层弹窗。
- 危险操作，例如删除任务、停止全部任务，应有明确反馈。

### 4.2 稳定性

- 任务执行过程中出现异常时，应停止当前任务并显示错误提示。
- 停止操作应优先于普通输入执行，避免任务失控。
- 应避免因单个任务配置错误导致应用整体崩溃。

### 4.3 性能

- 普通输入任务启动响应应尽量控制在 1 秒内。
- 运行日志追加不应明显影响任务执行。
- 本地任务数量较少时，任务列表应保持即时响应。

### 4.4 安全与边界

- 应明确提示用户仅在自己有权限操作的界面中使用本工具。
- 不提供绕过验证、隐藏执行或规避安全策略的功能。
- 默认不上传用户配置、坐标或执行日志。

### 4.5 可维护性

- 代码应按任务管理、步骤配置、执行引擎、日志记录、设置管理等职责拆分模块。
- 数据结构应预留版本字段，方便后续迁移。
- 关键执行逻辑应添加注释，说明状态流转和异常处理原因。

## 5. 数据结构草案

### 5.1 点击任务

- `id`：任务唯一标识。
- `name`：任务名称。
- `description`：任务描述。
- `enabled`：是否启用。
- `repeatCount`：任务重复次数。
- `startDelayMs`：任务开始前延迟时间。
- `steps`：输入步骤列表。
- `createdAt`：创建时间。
- `updatedAt`：更新时间。

### 5.2 输入步骤

- `id`：步骤唯一标识。
- `name`：步骤名称。
- `enabled`：是否启用。
- `actionType`：输入动作类型，包括鼠标点击和键盘按键。
- `x`：屏幕横坐标。
- `y`：屏幕纵坐标。
- `clickType`：点击类型。
- `keyName`：键盘按键名称。
- `keyPressCount`：键盘连按次数。
- `keyIntervalMs`：键盘连按间隔。
- `beforeDelayMs`：执行前等待时间。
- `afterDelayMs`：执行后间隔时间。
- `order`：执行顺序。

### 5.3 执行记录

- `id`：记录唯一标识。
- `taskId`：关联任务标识。
- `status`：执行结果。
- `startedAt`：开始时间。
- `endedAt`：结束时间。
- `message`：结果说明或错误说明。

## 6. 核心流程

### 6.1 创建任务流程

1. 用户点击“新建任务”。
2. 用户填写任务名称和基础配置。
3. 用户添加输入步骤。
4. 用户通过坐标采集或手动输入设置坐标。
5. 用户保存任务。
6. 系统将任务显示在任务列表中。

### 6.2 执行任务流程

1. 用户选择一个任务。
2. 用户点击“启动”或使用快捷键启动。
3. 系统进入倒计时或直接执行。
4. 系统按步骤顺序执行鼠标点击或键盘按键。
5. 用户可随时暂停、继续或停止。
6. 系统记录执行结果并展示日志。

### 6.3 坐标采集流程

1. 用户在步骤编辑区点击“采集坐标”。
2. 系统进入坐标采集模式。
3. 用户移动鼠标到目标位置。
4. 用户确认坐标。
5. 系统写入当前步骤。

## 7. MVP 验收标准

- 可以创建、编辑、删除和复制点击任务。
- 可以为任务配置多个输入步骤，包括鼠标点击和键盘单键连按。
- 可以采集鼠标坐标并写入步骤。
- 可以设置键盘按键、连按次数和连按间隔。
- 可以启动、暂停、继续和停止任务。
- 可以设置任务重复次数和步骤间隔。
- 可以查看最近执行日志。
- 应用重启后可以恢复已保存任务。
- 停止任务操作在任务运行中始终可用。

## 8. 后续扩展方向

- 支持任务导入和导出。
- 支持定时启动任务。
- 支持按窗口标题绑定任务。
- 支持图片匹配点击。
- 支持更多鼠标动作，例如拖拽、滚轮。
- 支持组合键和文本输入步骤。
- 支持任务分组和标签。

## 9. 已确认决策

- MVP 只支持 Windows。
- 技术栈采用 C# + WPF，保持 Windows 原生体验。
- 点击坐标首版使用屏幕绝对坐标，后续再支持窗口相对坐标。
- 必须支持全局快捷键。
- 数据存储采用 SQLite。
- 首版不支持多显示器专项适配。
- 首版不支持任务导入导出。
- UI 风格为优雅简洁，可参考透明毛玻璃感。
