# Click Assistant 移动端延伸规划

文档版本：v0.1.0
最后更新：2026-07-09
维护人：项目负责人
当前状态：Android v0.4.0 已实现简化多页面界面、组合任务和光标取点，待继续真机验证

本文档用于记录 Click Assistant 从 Windows 桌面端延伸到手机端的软件规划。移动端不是简单复用桌面端点击能力，而是需要重新评估系统权限、交互方式、执行安全和分发限制。

## 1. 规划结论

- 移动端第一阶段建议采用 Android-first（优先 Android）路线。
- Android 原型优先验证辅助功能服务（Accessibility Service）能否稳定执行点击、滑动和文本相关动作。
- iOS/iPadOS 暂不承诺跨应用自动点击能力，仅作为后续调研方向。
- 桌面端现有领域模型可以作为任务、步骤、执行日志和设置模型的参考，但 Windows 输入执行器不能直接复用到移动端。
- 移动端首版不追求与 Windows 功能完全一致，优先验证核心闭环：创建任务、配置步骤、授权辅助功能、执行任务、停止任务、记录结果。
- 当前已补充 Android 端需求草案、低保真原型和 `mobile/android/` 轻量原型工程。

## 2. 平台边界

### 2.1 Android

Android 提供 `AccessibilityService`，可在用户明确授权后观察界面变化，并通过 `dispatchGesture` 分发手势。该能力适合验证点击、滑动和简单流程自动化，但需要清晰说明用途，避免设计成绕过安全验证、隐藏执行或未经授权控制第三方应用。

Android 方向的核心约束：

- 用户必须在系统设置中手动开启辅助功能服务。
- 应用需要提供明确、可理解的权限说明。
- 点击坐标需要考虑屏幕尺寸、分辨率、方向、刘海区、导航栏和多窗口模式。
- 系统版本、厂商 ROM、省电策略和悬浮窗限制可能影响后台运行稳定性。
- Google Play 对辅助功能权限用途有审查要求，产品文案和功能边界必须谨慎。

### 2.2 iOS/iPadOS

iOS/iPadOS 第三方 App 运行在严格沙盒中，通常不能像 Windows 桌面工具一样跨应用注入鼠标、键盘或触摸输入。XCTest 等能力主要面向开发测试，不适合作为面向普通用户的跨应用自动点击产品能力。

iOS 方向的建议：

- 不作为第一阶段实现目标。
- 后续只探索“任务配置器”“远程控制桌面端”“快捷指令集成”等受限能力。
- 不在产品承诺中描述跨应用全局点击或自动输入，避免形成不可交付需求。

## 3. 产品形态

Android 首版建议定义为“手机端重复操作助手”，而不是 Windows 版的完整移植。

首版目标：

- 用户可以在手机上创建和编辑任务。
- 用户可以配置点击、等待、滑动和基础文本输入步骤。
- 用户可以通过辅助功能授权执行任务。
- 用户可以随时从应用内停止任务。
- 运行前必须展示安全确认。
- 执行结果写入本地日志。

首版不纳入：

- 绕过登录、验证码、支付确认或系统安全弹窗。
- 隐藏执行、无感后台执行或绕过系统权限提示。
- iOS 跨应用点击执行。
- 云同步、账号体系、多设备协作。
- 图像识别、OCR 或 AI 自动识别界面元素。

## 4. 技术路线

### 4.1 推荐路线

建议先用原生 Android 技术栈实现移动端可行性原型：

- Kotlin 作为主要语言。
- Jetpack Compose 构建界面。
- Room 或 SQLite 保存任务、步骤和执行日志。
- AccessibilityService 承担点击、滑动和基础文本相关动作执行。
- Foreground Service（前台服务）用于展示运行状态和停止入口。

当前仓库中的 `mobile/android/` 第一版工程为了降低原型验证成本，暂时采用 Java、Android framework View 和 `SharedPreferences`，只验证辅助功能授权、固定坐标点击、停止入口和本地任务保存。待该链路在真机验证通过后，再迁移到 Kotlin、Jetpack Compose 和 Room。

选择原生 Android 的原因：

- 辅助功能服务是 Android 平台核心能力，原生实现资料、权限模型和调试路径更清晰。
- 当前项目的真正难点在系统能力验证，而不是跨平台 UI 复用。
- 等 Android 原型验证通过后，再判断是否需要引入 .NET MAUI、Flutter 或共享任务格式。

### 4.2 可复用内容

Windows 端可复用或参考的内容：

- 任务模型：任务名称、描述、启用状态、重复次数、开始延迟。
- 步骤模型：动作类型、执行顺序、执行前等待、执行后间隔、启用状态。
- 执行引擎思路：启动、暂停、继续、停止、取消令牌和状态流转。
- 安全策略：执行前确认、停止优先、运行中锁定配置。
- 文档结构：需求、原型、架构、测试和发布说明的组织方式。

需要重新实现的内容：

- 点击执行器。
- 文本输入执行器。
- 坐标采集方式。
- 全局停止入口。
- 权限授权流程。
- 移动端界面和触控交互。

## 5. Android 原型范围

Android 第一版原型建议控制在 2 到 3 周内，以验证能力边界为目标。

### 5.1 原型功能

- 创建、编辑、删除任务。
- 添加点击步骤，支持坐标、点击次数、步骤等待和步骤间隔。
- 添加等待步骤。
- 添加滑动步骤，支持起点、终点和持续时间。
- 可选添加文本输入步骤，先验证在可编辑输入框中的可行性。
- 执行前安全确认。
- 开始、停止和基础执行状态展示。
- 本地保存任务和最近执行日志。

### 5.2 原型验收标准

- 能在 Android 真机上完成辅助功能授权。
- 能在用户授权后执行固定坐标点击。
- 能稳定执行包含点击和等待的简单任务。
- 能在运行中通过明显入口停止任务。
- 能记录成功、停止和失败结果。
- 失败时能给出可理解的错误说明。

## 6. 工程结构建议

如果新增独立 Android 工程，建议放在：

```text
mobile/
  android/
    app/
    docs/
```

如果后续需要共享任务格式，可增加：

```text
schemas/
  click-task.schema.json
```

共享格式建议以 JSON Schema（JSON 架构）定义，而不是一开始强行共享 C# 领域模型。这样 Windows、Android 和后续其他端都能独立实现执行器，同时保持任务导入导出的数据结构一致。

## 7. 数据结构方向

移动端任务格式应尽量与 Windows 端保持概念一致，但字段需要允许平台差异。

建议新增平台相关字段：

- `platform`：任务目标平台，例如 `windows`、`android`。
- `coordinateMode`：坐标模式，例如 `absolute`、`relative`、`normalized`。
- `screenProfile`：屏幕宽高、方向和密度等采集时环境信息。
- `gesture`：移动端手势类型，例如 `tap`、`swipe`、`longPress`。

首版 Android 可以先使用本地 SQLite 或 Room 表结构。等任务导入导出进入计划后，再统一沉淀跨端 JSON Schema。

## 8. 安全与合规

移动端必须比桌面端更强调边界提示：

- 首次开启辅助功能服务前，必须解释应用会执行什么能力。
- 每次执行任务前，必须展示任务摘要和停止方式。
- 不提供绕过验证码、支付确认、权限弹窗或系统安全机制的功能。
- 不默认上传用户任务、坐标、文本内容和执行日志。
- 不将应用描述为后台无感控制工具。
- 对高风险动作保留显式确认。

## 9. 实施步骤

建议按以下顺序推进：

1. 补充 Android 端需求草案和低保真原型。
2. 新建 `mobile/android/` 原型工程。
3. 实现任务和步骤的本地保存。
4. 接入 AccessibilityService，先验证固定坐标点击。
5. 增加等待、滑动和停止流程。
6. 补充执行前确认和执行日志。
7. 在 2 到 3 台 Android 真机上做兼容性验证。
8. 根据验证结果决定是否继续产品化、是否加入任务导入导出和跨端共享格式。

当前进度：第 1 到第 6 步已在源码层面实现（简化首页 / 任务库 / 任务编辑 / 执行日志页面、多任务库、任务复制与启用、点击 / 滑动 / 等待 / 文本输入多步骤、点击与文本组合模板、执行前确认、执行日志、执行快照、启动暂停继续停止、跨应用悬浮控制和可拖动光标坐标拾取），并兼容旧版单任务数据迁移。移动端明确排除键盘按键与组合键注入；Android v0.4.0 debug APK 可在本机构建，后续通过 USB 在真机验证执行稳定性与兼容性。
