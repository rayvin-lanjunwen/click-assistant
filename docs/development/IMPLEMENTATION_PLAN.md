# Click Assistant 编码实施计划

文档版本：v0.2.0
最后更新：2026-07-09
维护人：项目负责人
当前状态：首版实现已完成，已支持鼠标点击、键盘连按、执行前安全确认、快捷键自定义和核心自动化测试，待补组合键/文本输入、CI 和打包发布

本文档用于记录 Click Assistant 从设计阶段进入编码与实现阶段的实施计划。编码阶段应以“先骨架、再核心、后界面、最后接系统能力”为原则，保证每一步都能验证、能回退、能扩展。

## 1. 已确认编码决策

- 首版平台：仅支持 Windows。
- UI 框架：WPF，不使用 WinUI。
- 开发语言：C#。
- `.NET` 版本策略：使用本机已安装的稳定版本。
- 当前环境检查：已安装 .NET SDK `10.0.301`，安装位置为 `C:\Users\15884\.dotnet\`。
- PATH 状态：已将 `C:\Users\15884\.dotnet` 写入用户级 PATH；当前已打开的终端或 IDE 需要重启后才能直接使用 `dotnet` 命令。
- 数据库：SQLite。
- 数据库访问方式：轻量手写 SQL，不使用 Entity Framework Core（EF Core，实体框架核心）。
- 坐标模式：首版使用屏幕绝对坐标。
- 快捷键：必须支持全局快捷键，尤其是立即停止。
- 多显示器：首版不做专项适配。
- 任务导入导出：首版不做。

## 2. 编码阶段目标

编码阶段的目标不是一次性完成所有功能，而是逐步获得一个可运行、可验证、可维护的桌面应用。

阶段性目标：

- 建立标准 C# WPF 工程结构。
- 实现核心领域模型和基础校验。
- 实现 SQLite 数据库初始化和手写 SQL 数据访问。
- 实现任务管理、步骤配置和执行日志的应用服务。
- 搭建 WPF 主界面骨架。
- 实现执行引擎状态流转。
- 最后接入 Windows API 点击、坐标采集和全局快捷键。

当前完成情况：

- 已完成工程结构、领域模型、SQLite 数据访问、应用服务、WPF 主界面、真实鼠标点击执行、键盘连按执行、坐标读取、全局停止快捷键、执行前安全确认和停止快捷键自定义。
- 已创建 `tests/ClickAssistant.Tests/` 自动化测试工程，覆盖领域模型校验、执行引擎启动/暂停/继续/停止状态流转，以及 SQLite 仓储保存、读取和删除。
- 已完成 `dotnet build` 验证，当前构建结果为 `0` 个警告、`0` 个错误。
- 未完成 CI/CD、组合键/文本输入、应用打包发布、更完整的异常提示和 UI 自动化测试。

## 4. 工程结构计划

首版建议使用以下结构：

```text
src/
  ClickAssistant.App/
  ClickAssistant.Application/
  ClickAssistant.Domain/
  ClickAssistant.Infrastructure/
tests/
  ClickAssistant.Tests/
docs/
  planning/
  architecture/
  design/
  development/
```

说明：

- `ClickAssistant.App`：WPF 界面、ViewModel、资源和应用入口。
- `ClickAssistant.Application`：应用服务、用例编排和数据传输对象。
- `ClickAssistant.Domain`：实体、枚举、值对象和领域校验。
- `ClickAssistant.Infrastructure`：SQLite、Windows API、日志和外部系统能力。
- `ClickAssistant.Tests`：单元测试和集成测试。

## 5. 实施阶段

### 5.1 第一阶段：工程初始化

目标：

- 创建解决方案和项目结构。
- 配置基础工程文件。
- 确保空项目可以编译。

任务：

- 创建 `.sln` 解决方案。
- 创建 WPF 应用项目。
- 创建类库项目：`Application`、`Domain`、`Infrastructure`。
- 创建测试项目。
- 添加项目引用关系。
- 添加 `.gitignore` 和 `.editorconfig`。
- 执行首次 `dotnet build`。

验收标准：

- 解决方案可以成功编译。
- 项目目录符合架构设计。
- 空 WPF 应用可以启动。

### 5.2 第二阶段：领域模型

目标：

- 先完成不依赖 UI、不依赖数据库的核心模型。

任务：

- 实现 `ClickTask`。
- 实现 `ClickStep`。
- 实现 `ExecutionLog`。
- 实现 `ClickType`。
- 实现 `ExecutionStatus`。
- 实现基础校验规则。

验收标准：

- 任务名称、重复次数、延迟时间和步骤启用状态可以正确校验。
- 核心模型有单元测试覆盖。

### 5.3 第三阶段：SQLite 数据访问

目标：

- 建立本地持久化能力。

任务：

- 创建数据库初始化逻辑。
- 创建 `tasks`、`task_steps`、`execution_logs`、`app_settings`、`schema_migrations` 表。
- 使用参数化 SQL 实现任务保存、读取、删除。
- 使用参数化 SQL 实现步骤保存和排序。
- 使用参数化 SQL 实现执行日志写入。

验收标准：

- 应用首次启动可以创建数据库文件。
- 任务和步骤可以保存并重新读取。
- SQL 全部使用参数化方式，避免字符串拼接。

### 5.4 第四阶段：应用服务

目标：

- 在 UI 和基础设施之间建立清晰的应用层。

任务：

- 实现 `TaskService`。
- 实现 `StepService`。
- 实现 `ExecutionLogService`。
- 实现 `SettingsService`。
- 定义必要的数据传输对象。

验收标准：

- UI 不需要直接访问 SQLite。
- 应用服务可以完成任务创建、编辑、删除和复制。

### 5.5 第五阶段：WPF 界面骨架

目标：

- 先实现可用界面，不急于完成最终视觉效果。

任务：

- 创建主窗口三栏布局。
- 创建任务列表区域。
- 创建任务详情区域。
- 创建步骤列表区域。
- 创建运行状态区域。
- 创建任务编辑弹窗。
- 创建步骤编辑弹窗。

验收标准：

- 可以通过界面创建和保存任务。
- 可以通过界面添加和编辑步骤。
- 应用重启后可以恢复任务列表。

### 5.6 第六阶段：执行引擎

目标：

- 先跑通执行状态，不直接接真实点击。

任务：

- 实现执行状态机。
- 实现启动、暂停、继续和停止。
- 使用模拟输入执行器验证流程。
- 写入执行日志。

验收标准：

- 状态可以按设计流转。
- 立即停止可以优先响应。
- 任务执行结果可以写入日志。

### 5.7 第七阶段：Windows 系统能力

目标：

- 接入真实桌面能力。

任务：

- 实现当前鼠标坐标读取。
- 实现屏幕绝对坐标点击。
- 实现左键单击、左键双击、右键单击。
- 实现全局快捷键注册。
- 实现快捷键冲突提示。

验收标准：

- 坐标采集可以写入步骤。
- 执行任务可以真实点击。
- 全局立即停止快捷键可用。

### 5.8 第八阶段：视觉优化

目标：

- 在功能稳定后再完善 UI 质感。

任务：

- 调整主窗口视觉层级。
- 实现半透明或近似毛玻璃效果。
- 优化按钮、表格、输入框和状态标签样式。
- 强化立即停止按钮的危险状态。

验收标准：

- 界面优雅简洁。
- 毛玻璃效果不影响文字可读性。
- 运行状态和危险操作足够醒目。

## 8. 风险与应对

### 8.1 当前终端未刷新 PATH

风险：SDK 已安装，但当前已打开的终端或 IDE 可能尚未刷新用户级 PATH，导致直接输入 `dotnet` 仍提示命令不可用。

应对：

- 重启终端、VS Code 或 IDE。
- 如需立即执行，可临时使用绝对路径 `C:\Users\15884\.dotnet\dotnet.exe`。
- 重启后再次运行 `dotnet --version` 和 `dotnet new wpf -h` 验证。

### 8.2 毛玻璃效果实现成本偏高

风险：WPF 原生毛玻璃效果可能需要额外 Windows API 或兼容性处理。

应对：

- 首版可使用半透明面板、柔和阴影和浅色背景实现近似效果。
- 功能稳定后再逐步增强视觉效果。

### 8.3 全局快捷键冲突

风险：用户系统中已有程序占用快捷键。

应对：

- 注册失败时给出明确提示。
- 允许用户通过界面按钮继续操作。
- 后续扩展快捷键自定义。
