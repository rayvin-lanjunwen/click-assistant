# Click Assistant 风格指南

文档版本：v1.0.0
最后更新：2026-07-09
维护人：项目负责人

本文档用于记录 Click Assistant 项目的整体风格约定。后续新增内容时，应尽量保持表达统一、结构清晰，并优先遵循本文件中已有的规范。

对 `STYLEGUIDE.md` 的任何修改，都应在 `CHANGELOG.md` 中以 `docs:` 类型记录变更内容。

## 1. 语言表达风格

- 项目文档、说明文字、注释和面向用户的提示内容，默认采用中文表达。
- 文件名、命令、变量名、接口名、技术术语和行业通用表达可以保留英文或其他语言。
- 当英文或其他语言内容可能影响理解时，应在首次出现时补充中文说明。例如：`workflow`（工作流）、`scheduler`（调度器）。
- 表达应尽量准确、简洁、清晰，避免含糊、口语化过强或前后不一致的描述。
- 同类内容应使用一致的名称和表述方式，减少一个概念出现多种叫法的情况。

## 2. 编码风格

### 2.1 结构设计

- 代码应优先采用模块化、系统化、结构化的组织方式。
- 不同职责应尽量拆分到清晰的模块、类、函数或配置中，避免单个文件或函数承担过多逻辑。
- 代码结构应方便后续修改、扩展和维护，新增功能时优先延续现有架构和命名习惯。
- 可复用逻辑应适当抽象，但不要为了抽象而抽象；抽象应服务于清晰性、稳定性和可扩展性。

### 2.2 注释规范

- 每一块主要代码逻辑都应添加必要注释，说明该部分的用途、职责或关键流程。
- 重要代码行、关键判断、边界处理、异常处理和不直观的实现，应添加更明确的注释。
- 注释应解释“为什么这样做”或“这段逻辑负责什么”，避免只重复代码本身。
- 当代码逻辑发生变化时，应同步更新相关注释，避免注释与实际行为不一致。

### 2.3 命名规范

- 代码文件、目录、变量、函数、类、组件和配置项的命名应准确表达其用途。
- 命名尽量采用英文，或基于项目中已有规范进行延伸。
- 同类概念应保持命名风格一致，避免混用缩写、拼音、模糊词或过度简写。
- 命名应优先清楚，其次简短；不要为了短而牺牲可读性。
- 对已有标准或框架约定的名称，应优先遵循其官方或社区通用写法。

### 2.4 提交规范

- 提交信息应统一采用 Conventional Commits（约定式提交）格式，便于自动生成更新日志。
- 每条提交信息的格式为：`<类型>: <简短描述>`。
- 提交描述应简洁明确，说明本次提交完成的核心变更。
- 常用类型包括：
  - `feat`：新功能。
  - `fix`：Bug 修复。
  - `docs`：文档变更。
  - `style`：代码格式调整，不影响代码运行逻辑。
  - `refactor`：代码重构，既不是新增功能，也不是修复 Bug。
  - `perf`：性能优化。
  - `test`：测试相关变更。
  - `chore`：构建流程、依赖、辅助工具或项目维护类变更。
- 提交信息示例：

```text
feat: 添加暗黑模式支持
fix: 修复登录超时未跳转问题
docs: 更新 README 中的安装步骤
```

- 如有对应的 Issue 或 PR，可在提交信息末尾补充关联编号，例如：`fix: 修复支付回调重复问题 (#42)`。

### 2.5 分支命名规范

- 分支命名统一使用小写英文字母、数字和短横线，避免使用空格、中文或特殊符号。
- 功能开发分支使用 `feature/xxx`，例如：`feature/user-login`。
- Bug 修复分支使用 `fix/xxx`，例如：`fix/payment-timeout`。
- 文档或示例分支使用 `docs/xxx`，例如：`docs/api-example`。
- 紧急修复分支使用 `hotfix/xxx`，例如：`hotfix/critical-security-patch`。
- 禁止直接向 `main` 分支推送代码，所有改动必须通过 Pull Request（拉取请求）合并。

### 2.6 代码格式规范

- 缩进统一使用 4 个空格，禁止使用 Tab。
- 行宽建议不超过 120 个字符。
- 字符串默认使用双引号；如项目主流语言或框架已有明确约定，应优先遵循项目约定。
- 大括号换行风格：函数和类的大括号另起一行；控制语句（如 `if`、`for`）的大括号不换行。
- 具体格式配置以项目根目录下的 `.editorconfig` 和 `.prettierrc` 为准。
- 自动格式化工具应作为最终执行标准，手动编写代码时也应尽量保持同样格式。

## 3. 工作日志规范

- 存放位置：工作日志统一记录在项目根目录下的 `WORKLOG.md` 文件中。
- 排序方式：按日期倒序排列，最新日期放在最上方；同一天内有多条记录时，按时间倒序排列。
- 时间要求：每条工作日志的时间必须精确到分钟，格式为 `YYYY-MM-DD HH:MM`。
- 时间字段记录的是工作开始时间；如果同一时间段涉及多件工作，应分别记录各自的开始时间。
- 内容要求：记录应简洁、准确，只需概括本次做了什么，例如“修复登录接口超时问题”。
- 工作日志不得粘贴大段代码、完整报错堆栈，也不需要具体到代码行号。
- 表述方式应优先使用清晰的动宾结构，例如“补充更新日志规范”“调整项目执行规则”。
- 更新时机：每次执行 `git commit` 前，必须确认当天的 `WORKLOG.md` 已同步更新。

## 4. 更新日志规范

- 存放位置：更新日志统一记录在项目根目录下的 `CHANGELOG.md` 文件中，并与 GitHub Releases（GitHub 发布页面）保持同步。
- 排序方式：按版本号倒序排列，最新版本放在最上方。
- 时间要求：每个版本的发布时间必须精确到分钟，格式为 `YYYY-MM-DD HH:MM`，并标注在版本号后面，例如 `## v2.1.3 - 2026-07-08 23:50`。
- 版本号必须严格遵循 Semantic Versioning（语义化版本），格式为 `主版本号.次版本号.补丁版本号`，例如 `v2.1.3`。
- 主版本号加一：用于不兼容的 API 变更、架构重写，或移除长期支持的功能。
- 次版本号加一：用于新增向后兼容的功能。
- 补丁版本号加一：用于修复 Bug、性能优化或不改变功能边界的改进。
- 内容必须详细、具体，说明变更内容和影响范围，避免只写“优化体验”“修复问题”等模糊描述。
- 每条更新日志应使用标准化前缀分类描述，常用前缀包括：`新增:`、`变更:`、`修复:`、`优化:`、`docs:`、`移除:`、`安全:`、`破坏性变更:`。
- 涉及 `STYLEGUIDE.md` 的变更必须使用 `docs:` 前缀，保持与文档变更类型一致。
- 如果存在对应的 Issue 或 PR，应在条目中关联编号，例如 `（Issue #12，PR #34）`；如果没有对应编号，可以省略。
- 更新时机：每次正式发布版本或打 Git Tag（Git 标签）前，必须先更新 `CHANGELOG.md`，再发布。

## 5. 目录结构规范

- 项目源代码统一放在 `src/` 目录下。
- 测试代码统一放在 `tests/` 目录下，并尽量与 `src/` 保持相同的目录层级。
- 配置文件放在项目根目录，例如 `.eslintrc`、`prettier.config.js`、`.editorconfig`。
- 项目根目录只保留入口文档和治理文档，例如 `README.md`、`CHANGELOG.md`、`WORKLOG.md`、`RULE.md`、`STYLEGUIDE.md`。
- 规划与需求类文档统一放在 `docs/planning/` 目录下，例如需求规格说明、产品原型蓝图和范围说明。
- 架构与技术方案类文档统一放在 `docs/architecture/` 目录下。
- 设计、交互和界面说明类文档统一放在 `docs/design/` 目录下。
- 开发实施、工程初始化和编码计划类文档统一放在 `docs/development/` 目录下。
- 会议记录、调研材料和阶段性分析文档应按主题放入 `docs/` 下的对应子目录，不得长期堆放在项目根目录。
- 构建输出目录应添加到 `.gitignore`，不纳入版本管理，例如 `dist/`、`build/`。
- 新增目录前应先确认职责边界，避免创建含义重复或命名模糊的目录。
